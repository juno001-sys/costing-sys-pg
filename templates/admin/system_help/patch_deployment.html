Patch Deployment Workflow

Standard procedure for shipping any code change to PROD. This workflow was set as the team standard on 2026-04-20 — follow it for every change, even "trivial" ones. The cost of an unnecessary 5-minute DEV check is much lower than the cost of an unintended PROD bug.

Never push directly to main without DEV verification. Even for one-line text changes. The discipline matters more than any individual change.

Branch model

BranchEnvironmentAuto-deploy
main PROD (real customers) Railway → push to main triggers prod deploy
dev DEV (test-costing-sys.up.railway.app) Railway → push to dev triggers dev deploy
claude/xxx or feature branches Local only (no auto-deploy) —

The 5-step deploy procedure

1. Make the change in a feature branch

Work in a feature branch (or worktree). Commit small, atomic changes with descriptive messages. Don't include unrelated changes in one commit.

2. Apply to LOCAL + DEV simultaneously

  • Copy the changed files into the main repo path so user can test at localhost:5050
  • Push the changes to dev branch so Railway DEV picks them up

These two should happen as one batch — don't make the user wait between them. Local + DEV are both verification surfaces for the same change.

3. Smoke-test on Kurajika internal company

This is what makes くらじか自然豊農 (the internal account) valuable as a smoke-test customer. The breakfast staff use it daily, so:

  • It always has fresh data
  • Real-world UX issues surface immediately (operators will complain)
  • Patches that break login, master data, or daily entry get caught before they affect external customers

After pushing to DEV, log in to DEV as admin@example.com (or any test account) and exercise the feature you just changed. If you can't easily reach the feature in 30 seconds, your release-note email needs better instructions.

4. Wait for explicit "apply to PROD" approval

Sys admin (super_admin role) reviews on DEV. Never assume approval from a previous similar change.

The trigger phrase to look for in conversation: "apply to PROD", "ship it", "looks good", "go for it" — explicit consent in the same session.

5. Merge to main and push

After approval: 

git checkout main
git pull origin main
git merge <feature-branch> --no-ff -m "merge: <short description>"
git push origin main
git checkout dev    # return to default branch

Railway picks up the push to main within ~1 minute and deploys to PROD.

What if a deploy needs a DB migration?

Schema changes follow a slightly different rhythm because Railway doesn't run migrations automatically.

  1. Write the SQL in init/migrate_YYYYMMDD_description.sql — force-add it past .gitignore with git add -f init/migrate_*.sql
  2. Apply to DEV first — use the saved DEV DATABASE_URL (see ~/.claude/.../reference_db_credentials.md)
  3. Test on DEV — verify the new code paths work, AND that existing pages still render (additive migrations should never break anything; if something breaks, the migration isn't truly additive)
  4. On PROD approval: apply the same SQL file to PROD before merging code to main. Order matters — code that depends on a column will 500 if the column doesn't exist yet.
  5. Then merge code to main as usual.
Migrations should be additive whenever possible. Adding a column with a default, adding a table, adding an index — all safe to apply with traffic running. Renames, drops, type changes need more care (multi-step deploys: deploy code that handles both old + new schema → migrate → deploy code that uses only new).

Rolling back if something breaks PROD

Quickest path:

  1. Identify the bad commit on main: git log --oneline main
  2. Create a revert commit: git revert <sha>
  3. Push: git push origin main
  4. Railway redeploys with the revert in <1 minute

For DB migrations: write a reverse migration if needed. Don't manually DROP COLUMN on PROD without a migration file — audit trail matters.

PROD ↔ DEV data sync

When DEV's data drifts too far from PROD (after weeks of test pollution), refresh DEV from PROD.

Two scripts depending on whether we have paying customers:

StageUseWhy
Pre-customer (current state) Full mirror via pg_dump + schema-drop Only Kurajika data on PROD; safe to copy everything
Post-customer init/sync_internal_data_to_dev.py --apply Filters out customer data; deny-by-default classification refuses to run if a new table is unclassified

The filtered script is committed and ready. The day the first paying customer's data lands on PROD, switch to it. See the script's docstring for the full procedure.

Always delete the local pg_dump file after restore — it contains real PROD user emails + password hashes.
