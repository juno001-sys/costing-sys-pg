Onboarding a New Client Company

End-to-end checklist for setting up a new paying customer. Read top to bottom — each step builds on the previous.

Time estimate: ~15 minutes for the technical setup, plus an onboarding call/meeting with the client's Chief Admin.

Pre-flight: confirm the contract

Before touching the system, you should know:

  • Company legal name (in 漢字 / kana for Japan)
  • Number of stores they want to register
  • Tier they signed for: Entry / Standard / Premium
  • Free trial period (defaults: Entry 1 month, Standard/Premium 3 months)
  • Email address of their Chief Admin (the person legally responsible — usually F&B manager or owner)
  • Whether they want a different start date than today

Step 1 — Register the client company

Sales role required.

  1. Go to /admin/system
  2. Click + Register New Client Company
  3. Fill in:
    • Company name (legal name in their language)
    • First admin's email — this becomes the Chief Admin
    • First admin's name
    • Temporary password — generate something secure (12+ chars, mixed); you'll send it via separate channel
  4. Submit. Audit log records action='CREATE' on mst_companies.
Don't email the temporary password in the same message as the username. Send the username/URL by email and the password via LINE / phone / in-person. Standard security hygiene.

Step 2 — Set the contract tier & trial

  1. Back at /admin/system, find the new company in the 🏢 Client Companies table
  2. Click ⚙ Features
  3. Pick the contracted tier from the dropdown (Entry / Standard / Premium)
  4. Set Trial Ends to the agreed date (e.g., today + 90 days for Standard/Premium). Leave blank if no trial.
  5. Confirm the monthly fee (auto-fills from tier baseline; override only if you negotiated a custom rate)
  6. Payment method: invoice (only option for now)
  7. Save. Audit log records action='CONTRACT_CHANGE'.

After save, the system creates a contract row in sys_company_contracts and seeds sys_company_features with that tier's defaults.

Step 3 — Create stores

This is done by the client's Chief Admin (NOT by us), but verify the flow works before handing off.

  1. Log out of sys-admin. Log back in as the new Chief Admin (use the temporary password).
  2. Navigate to 店舗マスタ (Stores Master)
  3. Add at least one store — this lets you test that purchase entry, stock count, and reports all work for the company
  4. Log out.

Step 4 — Send onboarding email

Hand off to the client. Include:

  • Login URL (PROD URL)
  • Their email + temporary password (separate channel for the password)
  • Link to the in-app help: /help/concept_authority for role basics, /help/setup_suppliers + related setup chapters for first-week onboarding
  • Their trial end date (if applicable)
  • Your contact info as the assigned Customer Success owner

Step 5 — Verify on Client Health (next day or two)

Within 48 hours, check /admin/system/health:

  • The new company should appear (filter to 🔵 new if they haven't logged in yet)
  • Once they log in, status flips to 🟢 healthy
  • If still 🔵 after 7 days → onboarding stuck → reach out

Step 6 — Schedule the 30-day check-in

Calendar reminder for ~30 days after start date. At that point you'd:

  • Open their Client Health detail page
  • Print it (🖨 Print button) → save as PDF → attach to the check-in email
  • Note which features they're using vs untouched
  • Suggest 1–2 features they're not using yet that would help them

Common gotchas

SymptomLikely causeFix
Client logs in but sees no menu items Tier not set yet (defaults to Premium-permissive but if contract was set to Entry, only Entry features show) Verify on ⚙ Features screen that tier matches contract; ensure features marked enabled.
Lifecycle banner shows "trial ends in 0 days" immediately Trial end date set to today by mistake Edit on ⚙ Features and push the date forward.
Client can't add an Admin user Their first user is set as admin (correct), but only Chief Admin can promote others to Admin. Did the first user get the chief flag? Check sys_user_companies.is_chief_admin = TRUE for the first user. The migration backfill normally handles this, but verify on PROD.
