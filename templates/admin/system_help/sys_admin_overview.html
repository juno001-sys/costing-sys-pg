Sys Admin Roles & Permissions

Every sys admin has one of four roles. Roles are stored in sys_users.sys_role and gate access to specific screens. All sys admins still see the System Admin tab (the home with company list); the other tabs appear only for the relevant roles.

Default: when a new sys admin is created, they get sys_role = 'super_admin' automatically. The current Super Admin should change it to the appropriate function role afterwards.

The four roles

Role Can access Cannot access
super_admin Everything. Plus: only role that can change other sys admins' sys_role. The "founder" / "boss" role. —
engineer System Admin home, Client Health (technical signals: errors, slow requests), Developer Dashboard (system-wide perf metrics). Tier setting on companies, invoice management, sys-role assignment.
sales System Admin home, Client Health (engagement signals: logins, usage), Manage Features on each client (set tier, override feature toggles). Invoice management, Developer Dashboard, sys-role assignment.
accounting System Admin home, Invoices (issue, mark paid, generate monthly batch). Client Health, tier setting, Developer Dashboard, sys-role assignment.

Per-screen access matrix

Defined in utils/sys_roles.py: SCREEN_ROLES. Edit there if you add a new sys-admin screen.

ScreenAllowed sys roles
System Admin home (/admin/system) super, engineer, sales, accounting
Manage Features (set tier, override toggles) super, sales
Client Health (overview + drill-down) super, sales, engineer
Invoices super, accounting
Developer Dashboard (/dashboard/dev) super, engineer
Assign sys role (this dropdown on home) super only

How to change a sys admin's role

  1. Log in as a super_admin.
  2. Open /admin/system → scroll to 🛡 Sys Admins section at the bottom.
  3. Use the Sys Role dropdown next to the user's email. Selecting a value submits immediately.
  4. Change takes effect on the user's next request — no logout required (we read sys_role from DB on every request).
  5. Audit log records the change as action = 'SYS_ROLE_CHANGE' in sys_work_logs.
Lockout safety: there's no DB constraint preventing the last super_admin from demoting themselves. Be careful — if you do this and no one else is super_admin, you'd need direct SQL access to recover. Recommended: keep at least 2 super_admins.

Adding a new sys admin

Currently no UI form for this — sys admins are created via direct DB insert or by repurposing the operator-creation flow with is_system_admin = TRUE. A dedicated "+ Add Sys Admin" form is on the roadmap.

SQL recipe (until the form ships):

-- 1. Create the user (use a real password hash — see werkzeug)
INSERT INTO sys_users
  (company_id, email, name, password_hash, role, is_active,
   is_system_admin, sys_role, created_at, updated_at)
VALUES
  (1, 'newadmin@kurashima.asia', 'New Admin',
   'pbkdf2:sha256:...', 'admin', 1,
   TRUE, 'engineer',  -- pick: engineer / sales / accounting
   now(), now())
RETURNING id;

-- 2. Membership in Kurajika's internal company (id=1)
INSERT INTO sys_user_companies (user_id, company_id, role, is_active)
VALUES (<new_user_id>, 1, 'admin', 1);
